Agents

Designing agent workflows that do not fall apart

Agentic systems become useful when they can call tools, make decisions, and recover from messy real-world inputs. They become production-ready when those decisions are bounded by contracts, state, evaluation, and observability.

I do not think of an agent as a magic autonomous worker. I think of it as a workflow that uses an LLM for reasoning at specific decision points. That distinction matters. A workflow can be tested, traced, retried, constrained, and explained. A vague autonomous loop is much harder to trust.

When I design an agent workflow, my goal is to make every step explicit enough that the system can fail safely. The LLM can still plan, choose tools, summarize evidence, and decide what to do next, but the application owns the boundaries.

Start with the job, not the agent

The first question I ask is not, "What can the agent do?" It is, "What job should this workflow complete?" For a mortgage research copilot, the job might be to answer a question using approved documents, cite the evidence, and refuse if the evidence is not present. For a natural-language-to-SQL assistant, the job might be to generate a safe Athena query, validate it, execute it, and explain the result.

Once the job is clear, I can define the workflow states: collect input, retrieve context, select tool, execute tool, validate output, generate answer, and record trace. This makes the system easier to reason about than an open-ended loop.

Use tools with strict contracts

Tool calling is where many agent demos become fragile. If a tool accepts loose text and returns unstructured text, the next step has to guess what happened. I prefer typed tool contracts with clear input schemas, output schemas, and failure modes. 

{
  "tool": "search_documents",
  "input": {
    "query": "borrower risk trend in Q2",
    "filters": {
      "document_type": "risk_report"
    },
    "top_k": 8
  },
  "output": {
    "matches": [
      {
        "document_id": "housing-risk-report-q2",
        "chunk_id": "chunk-014",
        "score": 0.83,
        "excerpt": "..."
      }
    ]
  }
}

A typed contract does not remove the need for LLM reasoning. It makes the reasoning safer because the model is choosing among well-defined actions instead of improvising an integration protocol at runtime.

Make state visible

Agents need memory, but not all memory belongs in the prompt. I separate durable workflow state from conversational context. Durable state includes the original user request, selected tools, retrieved documents, validation results, retry counts, and final answer status. Conversational context is only the subset the model needs for the next decision.

This approach makes debugging much easier. If the final answer is wrong, I can inspect whether the system retrieved the wrong evidence, selected the wrong tool, passed a bad argument, ignored a validation error, or generated unsupported text.

Design failure handling before success handling

Production agent workflows fail in ordinary ways: a tool times out, a query returns no rows, a document search retrieves weak evidence, an API returns malformed JSON, or a model produces a response that does not match the required schema. These cases should be part of the design, not surprises found later.

  • Timeouts: every tool call should have a maximum runtime.
  • Retries: retry only when the failure is likely transient.
  • Fallbacks: use simpler paths when advanced tools fail.
  • Refusals: return a clear answer when evidence or permissions are insufficient.
  • Human review: require approval for high-impact or low-confidence actions.

A good agent does not always complete the task. Sometimes the correct behavior is to stop, explain what failed, and avoid making the situation worse.

Guardrails should be close to the risk

I do not rely on one giant system prompt as the only guardrail. Prompts help, but the most reliable guardrails live near the risky action. SQL safety belongs near SQL execution. Citation validation belongs near answer generation. Permission checks belong before tool execution. Cost limits belong around model and retrieval calls.

This is especially important in enterprise systems. The application should be able to say, "The model asked to do this, but the policy does not allow it."

Trace every run

Agent bugs are often sequence bugs. The final answer may look strange because of a decision made four steps earlier. That is why I want traces for each run: user input, model decision, tool call, tool result, validation result, retry, fallback, and final output.

A useful trace lets me answer practical questions quickly: Which tool was selected? What arguments were passed? How long did it take? What did it cost? Which evidence was used? Did validation pass? Did the final answer cite the evidence correctly?

Evaluate the workflow, not just the final answer

For agent workflows, final-answer scoring is not enough. I also evaluate the intermediate behavior. If the final answer is correct but the agent used the wrong tool or skipped a required validation step, that is still a defect.

  • Did the agent choose the expected tool?
  • Did the tool arguments match the task?
  • Did the workflow stop when required evidence was missing?
  • Did retries happen only for retryable failures?
  • Did the final answer reflect the tool output accurately?

This kind of evaluation makes prompt and model changes safer. It also makes the system easier to explain to reviewers, teammates, and stakeholders.

The practical lesson

The best agent systems I want to build are not unconstrained loops. They are engineered workflows with explicit state, typed tools, local guardrails, traceable decisions, and evaluation at multiple layers.

That design gives the LLM room to reason while giving the application enough structure to remain reliable. For enterprise AI, that balance is the difference between a clever demo and a system people can actually use.
