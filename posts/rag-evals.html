RAG Evaluation

How I evaluate citation-grounded RAG systems

A RAG system is not production-ready because it gives a good answer once. It is production-ready when I can measure whether the answer is grounded, reproducible, and safe to ship after every prompt, model, or retrieval change.

When I build a citation-grounded RAG application, I treat evaluation as part of the system design, not as a final QA step. The goal is simple: if the assistant answers a question, it should retrieve the right evidence, cite the right source, avoid unsupported claims, and fail clearly when the answer is not available.

The failure modes I care about first

RAG systems usually fail in predictable ways. The retriever may miss the right document. The prompt may ask the model to answer more confidently than the evidence supports. The answer may contain a correct sentence with the wrong citation. Or the system may answer a question that should have been rejected because the corpus does not contain enough information.

I evaluate those risks separately because a single overall score hides too much. A system with strong language quality but weak citation accuracy is dangerous in a financial, healthcare, legal, or enterprise analytics context.

My minimum evaluation set

I start with a small golden dataset before adding more automation. Each case includes the user question, expected facts, required source documents, acceptable citations, and notes about what the assistant should not say. 

{
  "id": "case-001",
  "question": "What changed in borrower risk this quarter?",
  "expected_facts": ["delinquency improved", "risk remains elevated in one segment"],
  "required_citations": ["housing-risk-report-q2"],
  "negative_claims": ["risk disappeared", "all segments improved"]
}

This format is intentionally plain. It can live in JSONL, run in CI, and be reviewed by engineers, analysts, or domain experts without requiring a special labeling platform.

The metrics I track

I separate RAG evaluation into retrieval, answer, citation, and operational metrics.

  • Retrieval recall: did the retriever bring back the documents needed to answer the question?
  • Groundedness: are the answer claims supported by the retrieved context?
  • Citation correctness: do the cited chunks actually support the sentence they are attached to?
  • Refusal quality: does the system decline when the corpus does not contain the answer?
  • Latency: how long does retrieval, reranking, generation, and validation take?
  • Cost: how many tokens and model calls does one successful answer require?

Negative tests are not optional

I like adding questions where the right behavior is to say, "I do not have enough evidence." These cases are especially useful because they catch the most damaging kind of hallucination: an answer that sounds reasonable but cannot be traced back to the corpus.

For example, if the documents only discuss national housing trends, the assistant should not invent county-level mortgage risk. A good RAG system should know the difference between missing data and inconvenient data.

Regression testing prompts and models

Every prompt change should run against the same evaluation set. Every model upgrade should do the same. I want to know whether a new prompt improved groundedness but hurt refusal quality, or whether a cheaper model saved cost but lost citation precision.

That is why my evaluation runner stores prompt version, model name, latency, token usage, score breakdown, and failure notes for every run. The score is useful, but the failure notes are where the engineering work happens.

What I would ship first

For an MVP, I would not try to build a perfect evaluator. I would ship a simple and explainable loop:

  • A JSONL golden dataset with 30 to 50 realistic questions.
  • Retrieval recall checks against required documents.
  • LLM-assisted groundedness scoring with human-reviewable notes.
  • Citation matching checks for required sources.
  • Latency and token cost tracking for every run.
  • A CI quality gate that fails when core metrics regress.

The practical lesson

RAG quality is an engineering system. Chunking, retrieval, reranking, prompting, generation, citation formatting, and refusal behavior all interact. Without evaluation, every change is a guess. With evaluation, the team can improve the system deliberately.

That is the standard I want for AI applications in enterprise environments: not just a convincing answer, but a measurable answer with evidence behind it.
