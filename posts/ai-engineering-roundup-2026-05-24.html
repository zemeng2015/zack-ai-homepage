The small product bet I can actually execute

I have a lot of possible AI project directions: mortgage data, Athena analytics, RAG evaluation, agents, and developer tools. The hard part is not finding ideas. The hard part is choosing a small enough bet that I can execute with about one hour a day.

My first instinct is to build something technically impressive. But a side project only becomes a product when someone else understands the pain, trusts the result, and is willing to use it in their workflow. That changes the selection criteria.

For the next 90 days, the best bet is not the biggest market. It is the project that connects my strongest background, a real buyer pain, a narrow deliverable, and a credible path to the first paid pilot.

Why not start with a broad AI platform?

A broad AI platform is attractive because it feels ambitious. It is also dangerous for a solo builder with limited time. It creates too many surfaces at once: UI, auth, billing, hosting, integrations, support, and vague positioning. That is a lot of work before learning whether anyone wants the result.

I need a smaller loop. Find a concrete pain. Deliver a measurable outcome. Ask for feedback. Improve. Repeat.

Why JTestGen is the right-sized bet

JTestGen is not the only project I could build, but it matches my conditions. I know Java and Maven. I understand enterprise codebases and test debt. I can explain the value to a Tech Lead. The result can be measured in coverage improvement and Maven pass status.

The first version does not need to be SaaS. It can be an open-source CLI plus a manual or semi-automated service: I run a coverage audit, target a few low-coverage classes, generate and repair tests, and submit a reviewable PR.

The first offer should sell a result

The first commercial version should not sell a dashboard or subscription. It should sell a result:

  • I identify 1-3 low-coverage Java classes in a Maven project.
  • I generate tests with AI assistance and run Maven validation.
  • I repair failures instead of handing over broken generated code.
  • I provide before and after coverage, changed files, and a PR summary.

A $100-$300 coverage audit is not the final business model. It is a learning instrument. If no one wants that concrete result, a larger product would probably fail too.

The trust path matters

I should not begin with cold sales to strangers. The first users should be familiar teams, former colleagues, or engineers who already trust my technical judgment. That lowers the sales cost and increases the chance of honest feedback.

Open-source maintainers can come next. Unknown small companies can wait until the offer, examples, and case study are sharper.

My 90-day proof

  • Run JTestGen against real reproducible Maven projects, not only toy examples.
  • Publish a clear benchmark page with failures and limitations included.
  • Define a simple coverage audit offer.
  • Contact 3-5 trusted Java teams or engineers.
  • Complete one real test improvement PR with measurable coverage improvement.

Conclusion

The right product bet for me is not the most exciting idea on paper. It is the smallest credible bridge between my current abilities and a real paid outcome. For now, that bridge is JTestGen: a narrow AI-assisted workflow for improving Java test coverage with evidence a Tech Lead can review.
